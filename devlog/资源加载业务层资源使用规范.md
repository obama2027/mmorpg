# 资源加载业务层资源使用规范

## 目的

本文用于规范业务层在使用 `AssetBundle` 资源框架时的加载、持有、释放和预热方式，避免资源泄漏、重复持有、无效加载和引用计数异常。

## 一、总原则

1. 正式环境中，资源与场景加载统一走异步接口。
2. 业务层可以放弃加载结果，但不应打断底层共享加载过程。
3. 谁发起加载，谁负责释放对应的资源句柄。
4. 业务层必须区分“资源对象”和“资源句柄”。
5. 预热持有与业务加载持有是两套独立引用，必须分别释放。

## 二、资源加载规范

### 1. 始终优先使用异步接口

业务层统一使用：

- `GameAssetService.Instance.LoadAssetAsync<T>(key)`
- `GameAssetService.Instance.LoadSceneAsync(key)`

不要在正式环境业务代码中使用同步加载接口。

### 2. 必须持有 `AssetHandle<T>`

正确示例：

```csharp
private AssetHandle<GameObject> _panelHandle;

_panelHandle = await GameAssetService.Instance.LoadAssetAsync<GameObject>("UI.Login.Panel");
```

不推荐示例：

```csharp
GameObject prefab = (await GameAssetService.Instance.LoadAssetAsync<GameObject>("UI.Login.Panel")).Asset;
```

原因：

- 只拿资源对象，不保存 `AssetHandle<T>`，后续无法正确释放。
- 容易造成 bundle 引用计数无法回收。

### 3. 资源句柄的释放责任必须明确

规则：

- UI 界面加载的资源，由 UI 界面自己释放。
- 模块管理器加载的资源，由模块管理器自己释放。
- 预热系统持有的资源，由预热系统释放。
- 场景资源由场景卸载流程统一释放。

## 三、UI 界面层规范

### 1. `await` 返回后必须先判活

因为界面在等待资源过程中可能已经关闭、销毁或切场景。

推荐模板：

```csharp
private bool _isClosed;
private AssetHandle<GameObject> _panelHandle;
private GameObject _panelInstance;

private async Task LoadPanelAsync()
{
    AssetHandle<GameObject> handle = null;

    try
    {
        handle = await GameAssetService.Instance.LoadAssetAsync<GameObject>("UI.Login.Panel");

        if (_isClosed || this == null || !gameObject)
        {
            handle.Release();
            return;
        }

        _panelHandle = handle;
        _panelInstance = Instantiate(_panelHandle.Asset, transform, false);
    }
    catch (System.Exception ex)
    {
        if (handle != null)
        {
            handle.Release();
        }

        UnityEngine.Debug.LogError(ex);
    }
}
```

### 2. 界面关闭时必须释放句柄

推荐：

```csharp
public void Close()
{
    _isClosed = true;

    if (_panelInstance != null)
    {
        Destroy(_panelInstance);
        _panelInstance = null;
    }

    _panelHandle?.Release();
    _panelHandle = null;
}
```

### 3. `OnDestroy()` 里做兜底释放

所有动态加载资源的 UI，建议在 `OnDestroy()` 中再次兜底调用释放逻辑。

## 四、预热使用规范

### 1. 预热引用和业务引用分开计算

例如：

- `PreloadGroupAsync("Login")` 持有一份引用
- `LoadAssetAsync<GameObject>("UI.Login.Panel")` 又持有一份引用

这两份引用要分别释放：

- 业务层 `handle.Release()`
- 预热层 `ReleaseGroup("Login")`

不能认为释放其中一个就够了。

### 2. 预热只适合阶段性资源

适合预热的资源：

- 登录界面资源
- 主城公共 UI
- 战斗公共特效
- 场景切换前的必要资源

不建议把所有资源长期预热常驻。

### 3. 模块退出时及时释放预热组

例如：

```csharp
GameAssetService.Instance.ReleaseGroup("Login");
await AssetBundleManager.Instance.UnloadUnusedAssetsAsync(true);
```

## 五、场景加载规范

### 1. 场景统一走场景 key

通过 `AssetAddressConfig` 维护场景 key，不要在业务层直接写场景路径字符串。

正确示例：

```csharp
await GameAssetService.Instance.LoadSceneAsync("Scene.MainCity");
```

### 2. 场景卸载后配合内存回收

例如：

```csharp
await GameAssetService.Instance.UnloadSceneAsync("Scene.Login", true, true);
```

### 3. Additive 场景必须明确卸载责任

如果业务层使用 `LoadSceneMode.Additive`，则必须在不再需要时主动调用卸载接口。

## 六、避免的错误写法

### 1. 只拿资源对象，不保存句柄

错误：

```csharp
var prefab = (await GameAssetService.Instance.LoadAssetAsync<GameObject>("UI.Login.Panel")).Asset;
```

### 2. 重复加载后忘记释放

错误：

```csharp
await GameAssetService.Instance.LoadAssetAsync<GameObject>("UI.Login.Panel");
await GameAssetService.Instance.LoadAssetAsync<GameObject>("UI.Login.Panel");
```

如果两次都未保存和释放 handle，会导致引用泄漏。

### 3. 界面关闭后仍继续使用加载结果

错误：

- 界面已关闭
- `await` 返回后仍 `Instantiate`
- 或继续绑定按钮与刷新 UI

### 4. 把预热资源和业务加载资源混成一份引用理解

错误认知：

- “已经预热过了，所以业务层不用保存 handle 也没关系”

正确理解：

- 预热是预热系统持有
- 业务自己加载，还是业务自己持有和释放

## 七、推荐使用习惯

1. 所有业务层加载都优先保留 `AssetHandle<T>` 成员变量。
2. UI、模块、系统分别管理各自加载出来的句柄。
3. `await` 返回后先判对象是否仍有效。
4. 模块退出后统一做一次 `UnloadUnusedAssetsAsync()`。
5. 使用 `Group` 管理模块级资源，不要手动零散释放过多 bundle。
6. 通过配置 key 加载资源，不直接在业务层硬编码 bundle 名和资源路径。

## 八、一句话规范总结

- 资源对象是使用结果，资源句柄才是生命周期控制入口。
- 谁加载，谁释放。
- 预热引用和业务引用分开算。
- 业务层可以放弃结果，但不要干预底层共享加载。
- 界面关闭后，异步返回结果必须先判活再处理。
